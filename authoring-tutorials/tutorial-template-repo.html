Tutorial Template Repo

Background

One of the primary objectives of the OSC is to braoden both knowledge and practice of open science, and the online tutorials we offer are a cornerstone of this initiative. In that regard, some tutorials were brought over by our staff members from previous research institutions, and several more were commissioned between approximately 2021 and 2025.

While the content from these tutorials was designed and/or adapted to fit in with the other tutorials to create a logical learning flow, the tutorials were not technically standardized. That is, the licenses, HTML/CSS formatting, citation styles, and so on were not consistent between projects. Towards the end of 2024 and through 2025 we worked to align our existing tutorials as much as possible on these issues, but this was a significant amount of work.

Wanting to avoid a repeat of this process in the future, we established a templated GitHub repository, lmu-osc/tutorial-template with all of the default settings and files we wish to have applied to our projects going forward. The core contents of this repo are generally intended to be prescriptive when creating new tutorials, outside of deciding what content should be included in the tutorial itself. Let’s explore this in more detail below.

tutorial-template Overview

Primary documentation for the tutorial-template repo is available in the README.md file in the repository itself, but we will also provide a quick overview here to help orient you to the structure and contents of the template.

Quick Overview

  • Purpose: A lightweight Quarto site template pre-configured for OSC tutorials. It provides a simple, accessible structure that can be copied and adapted to publish a small course or tutorial website.
  • Who it’s for: Team members, contractors, and others who want a quick and easy way to create a tutorial website that follows OSC standards for licensing, citation, and formatting.
  • License: Content is licensed under CC-BY-SA 4.0; code is licensed under CC0 1.0 Universal.
  • Live example: https://lmu-osc.github.io/data-anonymization/

Structure Overview

The top-level files and directories you will commonly interact with are:

File / Directory Purpose
_quarto.yml Site configuration (title, navigation, output settings, theme). Update the title:, repo-url:, and navigation links to match your tutorial.
index.qmd The site homepage / landing page. Contains a tutorial overview, estimated time to complete, and links to sections.
about.qmd About page with license, citation, and contributor information.
CITATION.cff Citation metadata (authors, affiliations, ORCIDs) in a machine-readable format.
LICENSE.md / LICENSE-CODE.md License texts for content (CC-BY-SA 4.0) and code (CC0 1.0 Universal) respectively.
lmu-osc-custom.scss Site-wide SCSS tweaks for branding and visual changes (e.g., navbar color #00883A).
styles.css Additional CSS classes for page-level content styling (e.g., .center, .img-border).
matomo-analytics.html Analytics snippet for visitor tracking. Managed by OSC staff; do not edit unless instructed.
assets/ Directory for images, downloads, and other static files.
topic-one/, topic-two/ Example topic folders, each containing .qmd pages. Replace or rename these to match your tutorial sections.
.github/workflows/ GitHub Actions automation workflows (see Automation below).
_site/ Generated output site files. Do not edit directly — this is produced when the site is built.
.filenameignore Patterns ignored by the automated filename check workflow.
.gitignore Files and directories ignored by Git.

Navigation is controlled through the sidebar: and navbar: sections in _quarto.yml.

Adapting the Template

The template is designed to be forked via GitHub’s “Use this template” button. Once copied, the typical adaptation workflow is:

  1. Update _quarto.yml — Change the site title, repository URL, and navigation links to point to your new repository.
  2. Edit index.qmd — Write your welcome message, tutorial overview, and any software prerequisites.
  3. Edit about.qmd — Add author/contributor names, acknowledgements, and set the citation metadata.
  4. Replace topic folders — Rename or replace topic-one/ and topic-two/ with sections matching your tutorial content. Each section folder should contain an index.qmd as a landing page and additional .qmd files for individual chapters.
  5. Add images and files — Place them in assets/ and reference with relative paths (e.g., ![](/assets/images/logo.png)).
  6. Adjust styling — Modify lmu-osc-custom.scss for site-wide changes or styles.css for page-level CSS classes.
  7. Update CITATION.cff — Fill in author names, ORCIDs, and affiliations so your tutorial can be properly cited.
  8. Preview locally (optional) — Run quarto preview in the project directory after installing Quarto.
  9. Publish — Run quarto publish gh-pages to activate the GitHub Pages deployment pipeline. After this initial setup, changes pushed to the main branch are automatically rendered and published.

GitHub Actions Automation

The template includes several automated workflows located in .github/workflows/:

Workflow Purpose Triggers
publish.yaml Renders the Quarto site and publishes it to the gh-pages branch (i.e., the live website). Push to main, weekly schedule, or manual
citation-check.yml Validates CITATION.cff to ensure metadata is well-formed. Push to main (when CITATION.cff changes) or manual
filename-check.yml Enforces filename rules (length, depth, allowed extensions). Push to main, pull requests, or manual
style.yaml Formats R and Quarto/R Markdown code using styler. Manual only (workflow_dispatch)

To run a workflow manually: open the repository on GitHub → Actions tab → select the workflow → click Run workflow.

Conventions

  • File names: Lowercase and kebab-case (e.g., my-topic/my-page.qmd).
  • Content: Use plain text and simple Markdown/Quarto. Avoid embedding complex web widgets unless needed.
  • Collaboration: Edit content on GitHub using branches and pull requests so teammates can review changes.
  • Licensing: All new tutorials should use the same dual-license model (CC-BY-SA 4.0 for content, CC0 1.0 for code) unless a specific exception is approved.
